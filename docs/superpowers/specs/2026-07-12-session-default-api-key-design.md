# Session 默认兑换凭证设计

## 目标

当用户仅使用 ChatGPT Session 账号兑换且没有填写外部 API Key 时，系统自动使用服务器配置的 Session 默认兑换凭证。普通账号兑换仍必须由用户填写 API Key。两类账号同时存在时，系统按凭证模式拆分请求，不能让普通账号借用 Session 默认凭证。

## 安全边界

- 默认凭证只保存在服务器环境变量 `SESSION_REDEEM_API_KEY` 中。
- 前端源码、浏览器存储、网络响应、日志和构建产物不得包含默认凭证明文。
- 用户填写的 API Key 优先级最高；存在用户秘钥时，不启用默认凭证。
- 只有显式标记为 `session` 的请求可以在 API Key 为空时请求服务器使用默认凭证。
- 普通账号请求缺少 API Key 时继续返回明确的配置错误。

## 请求模型

兑换代理请求增加凭证模式字段：

```json
{
  "credentialMode": "session"
}
```

服务器解析规则：

1. 请求携带非空 `apiKey`：使用用户秘钥。
2. `apiKey` 为空且 `credentialMode` 为 `session`：使用 `SESSION_REDEEM_API_KEY`。
3. 其他情况：返回 HTTP 400，提示填写外部 API Key。
4. Session 默认凭证未配置：返回 HTTP 500/配置错误，不回传凭证明文。

## 前端数据流

Session 解析结果已有 `sourceType: "session"`，普通账号为 `sourceType: "account"`。该标记需要保留到兑换任务行，并用于选择凭证模式。

### 提交与预检

- 仅 Session：API Key 为空时使用 Session 凭证模式完成预检与提交。
- 仅普通账号：API Key 为空时阻止提交并提示填写。
- 混合输入且 API Key 为空：只提交 Session 批次；普通账号保留在输入池并显示未提交原因。
- 混合输入且已填写 API Key：可以沿用当前统一批次，全部使用用户秘钥。

### 后续操作

用户填写了 API Key 时，状态查询、自动轮询、取消、重试和自动换号全部继续使用用户秘钥。API Key 为空时，这些操作必须根据当前任务行的 `sourceType` 分组：

- Session 任务使用 Session 凭证模式。
- 普通任务不发送代理请求，并显示缺少用户 API Key 的错误。
- 混合任务拆成两次代理调用，再将返回明细合并为现有统一结果结构。
- 自动换号继承新账号的 `sourceType`，不能沿用旧任务的凭证模式。

## 后端结构

- 在代理配置中读取 `SESSION_REDEEM_API_KEY`。
- 新增纯函数负责解析有效凭证，避免在各路由重复判断。
- `/api/redeem/submit`、`/status`、`/cancel`、`/retry` 使用相同凭证解析逻辑。
- 日志仅记录端点、批次数量、凭证模式，不记录 API Key。

## 错误处理

- 普通账号缺少用户 API Key：`请先填写外部 API Key`。
- Session 默认凭证未配置：`服务器未配置 Session 默认兑换凭证`。
- 混合输入部分未提交：成功提交 Session 批次，同时明确报告普通账号未提交数量。
- 某一凭证模式请求失败时，不覆盖另一模式已成功返回的数据；页面按对应任务显示错误。

## 测试与验收

- Session 请求在用户 API Key 为空时使用注入的服务器默认凭证。
- 用户 API Key 始终覆盖 Session 默认凭证。
- 普通账号空秘钥仍被拒绝。
- 混合账号空秘钥只提交 Session 行。
- 混合状态查询、取消和重试按来源拆分并正确合并结果。
- 自动换号使用替换账号的来源类型。
- 服务端日志和客户端构建产物不包含默认凭证明文。
- 完整单元测试、生产构建和本地浏览器冒烟测试通过。

## 非目标

- 不在前端展示或自动填充默认凭证。
- 不改变 Plus 查询接口的认证方式。
- 不让普通账号自动使用 Session 默认凭证。
- 不引入服务器持久化任务数据库。
